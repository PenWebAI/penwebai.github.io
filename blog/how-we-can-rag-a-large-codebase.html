How to RAG Large Codebases: Indexing a Lot of Code Files

Why simple text splitting fails for code, and how we use ASTs and Dependency Graphs to build smarter retrieval systems.

We hit a wall. We built a standard RAG pipeline for our security agent, fed it a million lines of Python, and asked it to find a vulnerability. It failed. Not because the model was dumb, but because we were feeding it garbage.

The "Ctrl+F" Problem

Standard RAG is essentially a fancy Ctrl+F. You chop documents into 500-word blocks, index them, and hope that when you search for "auth bypass," you get the right block.

But code isn't a flat text document. It's a living organism of dependencies. If you slice a function in half because you hit the 500-word limit, you've lobotomized the context. We realized that to make an AI truly understand code, we had to stop treating code like literature and start treating it like a graph.

Phase 1: The Map (ASTs & Graphs)

We threw out the text splitters. Instead, we built a parser that thinks like a compiler. We use Abstract Syntax Trees (ASTs) to understand the anatomy of the code.

When our system ingests a file, it doesn't see lines of text. It sees nodes. It identifies a class, captures its methods as children, and links them. Then, we build a Dependency Graph. If `auth.py` imports `hashing.py`, we forge a hard link between them.

This means when the agent looks at a function, it doesn't just see the code—it sees the entire nervous system connected to that code.

Phase 2: The Compass (Hybrid Retrieval)

We also realized that vector search (semantic similarity) is terrible at exact variable names. If you search for `user_id_v2`, a vector model might give you `customer_identifier` because they mean the same thing. But in code, `user_id_v2` is a specific variable, and if you miss it, the code breaks.

We implemented a hybrid system. We use BM25 for exact keyword precision ("Find me this exact variable") and Vector Embeddings for conceptual understanding ("Find me authentication logic"). Merging these two signals gave us the best of both worlds.

Phase 3: Making the LLM Shine (The Agentic Shift)

This was our biggest breakthrough. Initially, we tried to do all the retrieval work for the LLM. We'd find the "best" chunks and paste them into the prompt.

We were wrong.

We realized that pre-fetching context is a guessing game. We don't know what the LLM needs to solve a complex bug. So, we stopped guessing. We gave the LLM eyes and hands.

The Active Investigator

Instead of a static prompt, we built an Agent with access to a suite of custom tools that let it traverse the codebase dynamically.

The Toolset

  • inspect_ast(node_id) Allows the agent to "expand" a node. It can choose to look inside a function body or collapse it to see just the signature.
  • trace_dependency(symbol) The agent can click "Go to Definition." If it sees a function call it doesn't understand, it fetches the source code for that function instantly.
  • generate_embedding_weight(context) This is the magic. The agent can analyze a code block and assign importance weights to different parts. It can say, "This error handling block is noise, weight it 0.1," or "This validation logic is critical, weight it 1.5."

Now, when we ask the agent to find a vulnerability, it doesn't just read a static report. It explores.

It might start at the `login()` function. It sees a call to `verify_password()`. It uses `trace_dependency` to jump to that file. It notices a weird import, uses `inspect_ast` to check the module level variables. It builds its own context window, piece by piece, driven by its own reasoning.

This shift from "Passive RAG" to "Agentic Exploration" turned our system from a document search engine into a true autonomous engineer.
